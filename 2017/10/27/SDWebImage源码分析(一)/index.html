介绍

最近都在尝试阅读源码以此来弥补一下挥霍的时光，SDWebImage 可以说是一个开发必备的库了，主要提供了 UIImageView 和 UIButton 的异步下载以及缓存功能。之前就有人推荐过 SDWebImage 这个库，说是非常值得阅读学习，然而之前太过懒惰而且浮躁，难以静心去阅读代码。现在看了下，刚打开项目目录，就感觉整个文件目录非常的清晰，现在就来一点点分析下去！

打开工程目录，可以清晰的看到整个框架的结构：

下面就一个个目录来说！

SDWebImageCompat（兼容类）

Compat 顾名思义兼容性，所以整个类主要处理兼容性的问题，兼容 Apple 其他的设备，毕竟现在 Apple 也和 Android 一样，愈发生态多样性，除了手机上 iOS 系统，还要兼容 TV 和 Watch。

首先引入 #import <TargetConditionals.h> 该系统文件定义了一些宏使得我们可以区分各个平台

GC

SDWebImage不支持垃圾回收机制，垃圾回收(Gargage-collection)是Objective-c提供的一种自动内存回收机制。在iPad/iPhone环境中不支持垃圾回收功能。
当启动这个功能后，所有的retain,autorelease,release和dealloc方法都将被系统忽略。

FOUNDATION_EXPORT 定义常量

FOUNDATION_EXPORT UIImage *SDScaledImageForKey(NSString *key, UIImage *image);

FOUNDATION_EXPORT NSString *const SDWebImageErrorDomain;

FOUNDATION_EXPORT 定义常量在检测字符串d额值是否相等的时候更快，可以直接用 ‘==’ 来比较，比较的是指针地址。

dispatch_main_async_safe

为什么需要定义这个 safe 宏呢，直接使用 dispatch_main_async 大多数情况就可以了，但不绝对。如果在主线程执行非主队列调度的API，而这个API需要检查是否由主队列上调度，那么将会出现问题。

Calling an API from a non-main queue that is executing on the main thread will lead to issues if the library (like VektorKit) relies on checking for execution on the main queue.

UIImage SDScaledImageForKey(NSString key, UIImage *image)

根据 key 来修改图片尺寸，如果 是 gif 图则递归，另外采用 inline 内联函数替代宏（不需要预编译），可以提高效率，但是编译器并不一定会采用内联的方式调用，可能会接调用函数。

小结

我们可以灵活运用系统提供的宏，以及自己定义的宏来进行预编译，这样显得更加灵活，并且可以很好的做好兼容性的处理，同时在定义宏是更严峻一点可以先判断 #ifndef 避免重复定义。

Singleton, init, dealloc

一个优秀的框架，必然提供多样性的初始化的方法，多样的定制

这里主要有单例初始化，还可以自己选择缓存在自定义的文件名中或者路径中。另外这里用到了 Designated Initializer 系统宏，用来指定构造器。这里简单记录下（参考：iOS Designated Initializers : Using NS_DESIGNATED_INITIALIZER）：

  • 每个类的正确初始化过程应当是按照从子类到父类的顺序，依次调用每个类的Designated Initializer。并且用父类的Designated Initializer初始化一个子类对象，也需要遵从这个过程。

  • 如果子类指定了新的初始化器，那么在这个初始化器内部必须调用父类的Designated Initializer。并且需要重写父类的Designated Initializer，将其指向子类新的初始化器。

  • 你可以不自定义Designated Initializer，也可以重写父类的Designated Initializer，但需要调用直接父类的Designated Initializer。

  • 如果有多个Secondary initializers(次要初始化器)，它们之间可以任意调用，但最后必须指向Designated Initializer。在Secondary initializers内不能直接调用父类的初始化器。

  • 如果有多个不同数据源的Designated Initializer，那么不同数据源下的Designated Initializer应该调用相应的[super (designated initializer)]。如果父类没有实现相应的方法，则需要根据实际情况来决定是给父类补充一个新的方法还是调用父类其他数据源的Designated Initializer。比如UIView的initWithCoder调用的是NSObject的init。

Cache clean Ops

清除缓存操作

  1. 清空内存：- (void)clearMemory;
  1. 清空磁盘：- (void)clearDiskOnCompletion:(nullable SDWebImageNoParamsBlock)completion;
  1. 清空过期数据: - (void)deleteOldFilesWithCompletionBlock:(nullable SDWebImageNoParamsBlock)completionBlock;

这里主要有两个指标：文件的缓存有效期及最大缓存空间大小。文件的缓存有效期可以通过maxCacheAge属性来设置，默认是1周的时间。如果文件的缓存时间超过这个时间值，则将其移除。而最大缓存空间大小是通过maxCacheSize属性来设置的，如果所有缓存文件的总大小超过这一大小，则会按照文件最后修改时间的逆序，以每次一半的递归来移除那些过早的文件，直到缓存的实际大小小于我们设置的最大使用空间。

AutoPurgeCache

这里内存缓存使用了 NSCache，NSCache 在收到内存警告的时候会释放自身的一部分资源, 平时基本没有使用过这个类，和字典类似。不同之处在于：

  1. NSCache 具有自动删除的功能，以减少系统占用的内存；

  2. NSCache 是线程安全的，不需要加线程锁；

  3. 键对象不会像 NSMutableDictionary 中那样被复制。（键不需要实现 NSCopying 协议）。

注册通知，当收到内存警告的时候，y移除所有内容、

定义了一个内联函数，用来计算图片缓存大小, 根据不同平台设置不同缓存大小。

小结

这一篇的话，主要就介绍的兼容处理以及缓存，缓存的话分为内存缓存和可选的磁盘缓存，内存缓存通过 NSCache 来处理。我们通常用使用 NSCache 来临时存储短时间使用但创建昂贵的对象。重用这些对象可以优化性能，因为它们的值不需要重新计算。另外一方面，这些对象对于程序来说不是紧要的，在内存紧张时会被丢弃。磁盘缓存的处理则是使用NSFileManager对象来实现的。图片存储的位置是位于Cache文件夹。另外，SDImageCache还定义了一个串行队列，来异步存储图片。本来是打算整个源码分析是放在一篇写的，但是越写到后面发现内容太多，看来得另开讲几篇文章来写的，毕竟还得考虑点可读性~~.
